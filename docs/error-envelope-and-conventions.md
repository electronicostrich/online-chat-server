# Error Envelope and API Conventions
## Online Chat Server

## 1. Purpose

This document defines the wire-level conventions that every HTTP response and WebSocket event must follow. It is the companion to `docs/api-and-events.md`:

- `api-and-events.md` lists the endpoints and events.
- This document specifies the shape every envelope takes.

Together with `packages/shared-schemas/src/schemas/envelopes.ts` (TypeBox source), these shapes are the single authority. AI-written handlers MUST import the shared envelope schemas rather than hand-constructing response bodies.

## 2. HTTP response envelopes

### 2.1 Success (single resource)

```json
{
  "data": { ... }
}
```

Rules:
- Every successful JSON response has a top-level `data` key.
- `data` is either an object (for single resources) or an array (for collections).
- No other top-level keys are permitted on success responses (see pagination below for the collection exception).

### 2.2 Success (paginated collection)

```json
{
  "data": [ ... ],
  "pagination": {
    "nextCursor": 104,
    "hasMore": true,
    "limit": 50
  }
}
```

Rules:
- `nextCursor` is the cursor to pass as the next `beforeSequence` or opaque `cursor` parameter, depending on endpoint.
- `nextCursor` is `null` when `hasMore === false`.
- For sequence-paginated chat history endpoints, `nextCursor` is an integer. For opaque-paginated listings (e.g., public room catalog), `nextCursor` is a string.

### 2.3 Success (no body)

```json
{ "data": { "ok": true } }
```

Avoid empty `204 No Content` responses; return `200 { "data": { "ok": true } }` instead. Rationale: consistent client parser, easier to extend with fields later.

### 2.4 Error

```json
{
  "error": {
    "code": "ROOM_BANNED",
    "message": "User is banned from this room.",
    "details": { ... },
    "traceId": "abc123..."
  }
}
```

Rules:
- `code` is one of the values in `api-and-events.md` §4.5. Must also appear in `packages/shared-schemas/src/constants/error-codes.ts`.
- `message` is a short, human-readable string. It SHOULD be safe to show to end users but is not localised. The frontend is responsible for displaying a localised string keyed by `code` when UX demands it.
- `details` is an optional object. For `VALIDATION_ERROR`, it contains `{ "fieldErrors": { "fieldPath": "reason" } }`. For other codes, its contents are documented per endpoint.
- `traceId` is always present. In dev it comes from the request-id plugin; in production it comes from the observability middleware.

**Clients never receive both `data` and `error` in the same response.** Success and error are mutually exclusive.

### 2.5 HTTP status mapping

See `docs/api-and-events.md` §4.5. That table is binding. Implementations MUST NOT return a status not listed there for a given code.

## 3. WebSocket envelopes

### 3.1 Event envelope (server → client)

```json
{
  "eventId": "uuid-v4",
  "type": "message.created",
  "occurredAt": "2026-04-18T13:00:05Z",
  "payload": { ... }
}
```

Rules:
- `eventId` is a v4 UUID generated by the server. Clients deduplicate on this field (see `api-and-events.md` §7.2).
- `type` is one of the values enumerated in `api-and-events.md` §6.4 and mirrored in `packages/shared-schemas/src/schemas/events.ts`.
- `occurredAt` is the server's UTC timestamp. Clients MUST NOT use `occurredAt` for ordering; authoritative order is `(chatId, sequence)` from the payload.
- `payload` is specific to the event type. Every payload has a TypeBox schema.

### 3.2 Command envelope (client → server)

```json
{
  "id": "client-generated-id",
  "type": "chat.subscribe",
  "payload": { ... }
}
```

Rules:
- `id` is a client-generated string, unique per connection. Used to correlate server replies (including `sync.response`).
- `type` is one of the values enumerated in `api-and-events.md` §6.2.
- `payload` is specific to the command type.

Unknown `type` values are rejected with an error event:

```json
{
  "eventId": "uuid-v4",
  "type": "command.error",
  "occurredAt": "...",
  "payload": {
    "replyToCommandId": "client-generated-id",
    "code": "VALIDATION_ERROR",
    "message": "Unknown command type: 'foo.bar'"
  }
}
```

### 3.3 Server-initiated close frames

When the server decides to close a WebSocket (session revoked, buffer overflow, protocol violation), it sends a close frame with a code from this table:

| Close code | Meaning | Client response |
|---|---|---|
| `4401` | Session revoked | Redirect to login |
| `4413` | Outbound buffer overflow | Reconnect, then send `sync.request` |
| `4400` | Protocol violation (malformed command) | Log; reconnect after 5s |
| `4429` | Rate limited | Reconnect after 30s with backoff |

The 4000-range codes are application-defined, consistent with RFC 6455.

## 4. Request shape conventions

### 4.1 Content type

- REST request bodies: `application/json` unless the endpoint is `multipart/form-data` (attachment uploads only).
- REST response bodies: `application/json` unless the endpoint returns a binary stream (attachment downloads only).
- WebSocket frames: text frames carrying JSON.

### 4.2 IDs

All resource identifiers are version-4 UUIDs unless otherwise specified. They appear as strings on the wire.

### 4.3 Timestamps

All timestamps are ISO-8601 UTC strings with explicit `Z` suffix: `"2026-04-18T13:00:05.123Z"`. Milliseconds are included when meaningful.

### 4.4 Integers

Integer IDs (`sequence`, `headSequence`, `lastReadSequence`) are JSON numbers, NOT strings. Maximum value fits in `Number.MAX_SAFE_INTEGER` (2^53 - 1), which is ~9 quadrillion — far beyond the practical message count of any chat.

### 4.5 Nulls vs missing

- A field that is explicitly nullable (documented in its schema as `Type.Union([..., Type.Null()])`) uses `null` on the wire.
- A field that is optional (absent in a partial response) is simply omitted.
- Clients MUST NOT treat `null` and missing as equivalent without checking the schema.

## 5. Pagination

Sequence-paginated endpoints (`GET /chats/{chatId}/messages` and similar):
- Follow `docs/api-and-events.md` §4.3.
- Response envelope: `data: [...], pagination: { nextCursor: number | null, hasMore, limit }`.

Opaque-paginated endpoints (`GET /rooms/public`):
- Accept `cursor` (string) and `limit` (integer).
- Response envelope: `data: [...], pagination: { nextCursor: string | null, hasMore, limit }`.
- The cursor value is opaque — clients MUST NOT parse or construct it. The server defines the format (typically a base64-encoded row position).

## 6. Idempotency

For state-changing requests where retry could be harmful (message send, invitation accept):

- Clients MAY send an `Idempotency-Key` HTTP header with a UUID.
- Servers that recognise the header MUST deduplicate against it for at least 24 hours.
- For endpoints that do not need idempotency (e.g., `POST /auth/logout`), the header is ignored.
- For WebSocket commands (`message.send`), the `id` field in the command envelope serves the same purpose within the lifetime of the socket.

Which endpoints honour `Idempotency-Key` is documented per-endpoint in `api-and-events.md`.

## 7. Rate limits

Rate-limited responses carry:

- Status `429`
- Error code `RATE_LIMITED`
- Header `Retry-After: <seconds>`
- Error `details.resetAt: ISO-8601 timestamp`

Default limits (per session):
- Authentication endpoints: 10 req/min
- Message send: 60 req/min
- Attachment upload: 10 req/min

These limits are configuration, not contract. Changes MUST update this document and, if they affect behaviour clients must implement, the relevant ACs.

## 8. CORS and CSRF

- CORS: `Access-Control-Allow-Origin` is set from `ALLOWED_ORIGINS` (see `runtime-and-environment.md` §6.1). `Access-Control-Allow-Credentials: true`.
- CSRF: every state-changing REST request (non-GET, non-HEAD) MUST carry an `X-CSRF-Token` header that matches the `csrf_token` cookie set at login. Failure → `CSRF_FAILED` (403).
- WebSocket handshakes validate the `Origin` header against `ALLOWED_ORIGINS` and the session cookie. No CSRF token is required at the WS handshake because the browser does not attach cookies to cross-origin WS handshakes unless same-site allows it.

## 9. Validation errors

When request validation fails, the response is:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": {
      "fieldErrors": {
        "/email": "must match email pattern",
        "/password": "must be at least 12 characters"
      }
    },
    "traceId": "..."
  }
}
```

- `fieldErrors` keys are JSON-pointer paths.
- The values are short English strings. Clients localise using `error.code` plus the field path if they want custom copy.

This shape is emitted automatically by the Fastify TypeBox error translator in `apps/api/src/plugins/errors.ts`. Handlers MUST NOT hand-roll validation error responses.

## 10. What this document enforces

- Every endpoint in `apps/api/src/modules/*/routes.ts` registers a response schema that produces one of the envelopes in §2.
- Every WebSocket event publisher uses the event envelope helper from `apps/api/src/realtime/events.ts`.
- Every client fetch wrapper (`apps/web/src/api/`) parses the `data` / `error` split and normalises errors before passing to TanStack Query.
- The Fastify error translator plugin ensures unexpected exceptions are converted to `INTERNAL_ERROR` envelopes rather than leaking stack traces.

Drift between this document and the TypeBox envelope schemas is a CI-blocking error (via `doc-consistency.yml`).
