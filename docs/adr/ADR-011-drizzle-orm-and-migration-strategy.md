# ADR-011: Drizzle ORM with SQL-First Forward-Only Migrations

- **Status**: Accepted
- **Date**: 2026-04-18

## Context

The data model (see `docs/data-model.md`) is relational, carefully constrained, and sensitive to subtle schema drift (uniqueness on normalized names, FK cascade behaviors, retention/deletion policy). Schema correctness is part of the product: a missing `ON DELETE CASCADE` or an enum that loses a value silently is a production-shaped bug.

The project needs a persistence layer that:

1. **Produces explicit, reviewable SQL migrations.** Black-box migration generation (where the ORM decides how to move between states) is rejected — the PO must be able to read a migration and understand what it does.
2. **Keeps the TypeScript schema definition and the database schema in lockstep.** Drift between the two is a CI failure, not a runtime surprise.
3. **Does not abstract away PostgreSQL features we depend on** (unique partial indexes on `normalized_name`, `inet` type, generated columns, `ON DELETE SET NULL`, etc.).
4. **Is quick for AI to write correctly** given the data-model doc.

Two candidates are commonly reached for: Prisma and Drizzle.

- **Prisma** hides SQL behind its own DSL and auto-generates migrations. The generated migrations can surprise (especially around enums and cascades), and the DSL is another thing to learn and keep in sync.
- **Drizzle** is SQL-first: TypeScript code *is* the schema, and migrations are generated as plain SQL files you review and commit. The query builder is thin and produces readable SQL.

## Decision

Use **Drizzle ORM** with **Drizzle Kit** for migrations.

### Rules

1. **Schema location**: `apps/api/src/db/schema/` — one file per entity (e.g., `users.ts`, `messages.ts`) exporting the Drizzle `pgTable` definitions. An `index.ts` re-exports everything.
2. **Migration location**: `apps/api/drizzle/` — every migration is a numbered SQL file generated by Drizzle Kit and committed to Git.
3. **Migration generation**: `pnpm --filter api db:generate` runs `drizzle-kit generate`. The generated SQL is reviewed and hand-edited if necessary before committing.
4. **Migration application**: `pnpm --filter api db:migrate` runs `drizzle-kit migrate` against the configured database URL. In Compose, this runs as an init step on API startup (idempotent).
5. **Forward-only**: no `down` migrations. Rolling back a deployment rolls forward with a new migration, not a reverse of the old one. Rationale: reverse migrations are rarely tested, usually wrong, and create a false sense of safety.
6. **Raw SQL allowed when needed**: for features Drizzle doesn't express cleanly (custom constraints, triggers), use `sql.raw()` inline or commit a `.sql` migration fragment that Drizzle will apply in order. Never run ad-hoc SQL from application code outside the schema/migration flow.
7. **Every FK cascade behavior** documented in `docs/data-model.md` §7.7 MUST be reflected in the Drizzle schema via `references(…, { onDelete: '…' })`. A CI check diffs the expected behaviors against the schema and fails on mismatch.
8. **Transactions**: the application uses Drizzle's `db.transaction(async (tx) => { ... })` for any operation that touches more than one row. Section 8 of `docs/data-model.md` lists the required transaction boundaries; each MUST be implemented with an explicit `tx`.

### Enum handling

Drizzle's `pgEnum` is acceptable. However, enums require care across migrations:

- **Adding values**: always appended, never reordered (`ALTER TYPE ... ADD VALUE 'x'`).
- **Renaming values**: forbidden. Create a new enum, migrate rows, drop old.
- **Removing values**: forbidden in place. Create a new enum, migrate rows, drop old.

The migration file for any enum change MUST include a comment block stating the transition and why.

### Schema drift detection (CI)

Drizzle Kit's actual behavior must be respected here: `drizzle-kit generate` compares the TypeScript schema to the prior migration snapshot in `apps/api/drizzle/meta/_journal.json`, not to a live database. The `schema-drift.yml` workflow therefore performs three orthogonal checks, not one:

1. **Schema ↔ latest migration check.** Run `drizzle-kit check` (which validates the journal integrity and the schema-vs-snapshot consistency). Fails if the journal is corrupt, if a migration file was hand-edited after generation, or if any committed migration contradicts another.

2. **Generator round-trip check.** Run `drizzle-kit generate` in a detached temp directory pointed at the repo's schema files. Fail if the generator produces any new migration file — this indicates the committed migrations do not fully express the current TypeScript schema (i.e., someone edited schema code without running `db:generate`).

3. **Fresh-DB apply round-trip.** Spin up a fresh Postgres service container. Apply all committed migrations via `drizzle-kit migrate`. Then connect to the live DB and use Drizzle's introspection (or `pg_dump --schema-only` compared against a snapshot) to confirm the resulting live schema matches the expected shape. Fail if introspection reveals a column/constraint/index that exists in code but not in the DB, or vice versa.

Check (1) catches journal tampering. Check (2) catches the single most common mistake — editing schema code without running `db:generate`. Check (3) catches the rarer case where a hand-edited migration succeeds at apply but diverges from what the schema code declares.

The workflow implementation lives in `scripts/schema-drift-check.ts` (see `docs/script-specs.md`). If any of the three checks fails, CI blocks merge.

### Seeding and fixtures

- Seed scripts live in `apps/api/src/db/seed/` and use Drizzle's query builder, not raw inserts.
- Playwright tests seed via a test-only Fastify route (`POST /__test/seed`) that is registered ONLY when `NODE_ENV=test`. This avoids shelling out and keeps seeding inside the transactional boundary.
- There is no fixture data committed to the repo outside the seed scripts.

## Consequences

### Positive

- Migrations are plain SQL in Git. PO can read them, dev leads can review them on GitHub as if they were application code.
- Schema drift is a CI failure, not a production bug.
- Cascade behaviors documented in `data-model.md` map directly to `references(... { onDelete })` calls — the doc-to-code distance is minimal.
- AI-friendly: Drizzle schemas look like TypeScript objects, and the code for a new entity is easy to pattern-match from an existing one.

### Negative

- No automatic relational query building (Drizzle has a `relations` helper for this; usage is optional). Some "find user with their rooms and bans" queries require explicit joins. Acceptable given the simplicity of the data model.
- No down-migrations means early-schema mistakes are harder to undo. Mitigated by review-before-merge discipline and by the fact that MVP is local-only — there is no production database at risk.
- Drizzle's tooling is newer than Prisma's. Pinning exact versions and enabling `schema-drift.yml` catches regressions introduced by Drizzle updates.

## Enforcement

- `docs/ai-development-guardrails.md` → "Any new column, FK, or index requires (a) schema code change, (b) migration file, (c) documentation update in `docs/data-model.md`."
- `docs/ci-pipeline.md` → `schema-drift.yml` blocks merge on drift.
- ESLint `no-restricted-imports` bans direct `pg` client usage outside `apps/api/src/db/` to prevent ad-hoc SQL.

## Review triggers

- Drizzle introduces a breaking change that materially increases migration authoring cost.
- The project adopts a read-replica or sharded Postgres setup that requires a different migration strategy.
- A requirement emerges for online schema changes that Drizzle's forward-only model cannot serve (unlikely pre-production).
